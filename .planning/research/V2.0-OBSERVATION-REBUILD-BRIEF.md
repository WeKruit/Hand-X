# Hand-X v2.0: Observation Layer Rebuild

## Research Brief for Team Discussion

**Date:** 2026-04-02
**Branch:** `feat/v2.0-observation-layer-rebuild`
**Confidence:** HIGH (grounded in codebase analysis, CDP docs, academic papers, and industry tools)

---

## TL;DR

We're replacing the observation layer — the part that understands what's on a web page. The action layer (filling forms, clicking buttons) stays. The problem isn't *how* we act, it's *what we see*.

**The recommended approach:** A three-layer pipeline that extracts raw DOM data, interprets it semantically in Python (not JS), and tracks state changes across steps. Zero LLM cost for observation. Builds on top of browser-use's existing CDP infrastructure. No new dependencies.

---

## 1. Why We're Doing This

### The Core Problem

The current `field_extractor.py` injects 500+ lines of JavaScript to traverse the DOM, classify field types, resolve labels, and group fields — all in one opaque blob. This causes:

| Problem | Example | Root Cause |
|---------|---------|------------|
| **Non-deterministic** | Same page produces different field lists on different runs | Timing-dependent: async-loaded content, DOM render order, transient widget states |
| **Non-generic** | Can't tell "Work Experience" from "Work History" | Grouping uses literal `(field_type, label, section)` matching — no semantic understanding |
| **Hallucinations** | Agent fills wrong field, selects wrong country | Gemini Flash overwhelmed by raw DOM context — too many fields, too many options |
| **State-blind** | Agent loops on already-selected checkboxes | Can't detect state changes in custom widgets (Workday, Oracle) |
| **Grouping failures** | Multiselect options treated as separate questions | DOM siblings ≠ semantic siblings — the DOM doesn't encode "these belong together" |

### What Works Fine (Don't Touch)

- **Action layer** — `domhand_fill`, fill strategies for selects/combos/radios/dates/files
- **Fill executor** — per-widget-type fill logic
- **Verification engine** — post-fill readback and verification
- **Platform guardrails** — system prompts with per-ATS hints
- **JSONL protocol** — Desktop ↔ Hand-X communication
- **HITL flow** — pause/review/resume

## 2. The Recommended Architecture

### Three-Layer Synchronous Pipeline

```
┌─────────────────────────────────────────────────┐
│                    Observer                       │
│                                                   │
│  ┌──────────┐   ┌──────────────┐   ┌───────────┐ │
│  │  Layer 1  │──▶│   Layer 2    │──▶│  Layer 3  │ │
│  │    Raw    │   │   Semantic   │   │   State   │ │
│  │ Extractor │   │ Interpreter  │   │  Tracker  │ │
│  └──────────┘   └──────────────┘   └───────────┘ │
│       │                │                  │       │
│  RawPageSnapshot  SemanticPageModel  ObservationState
└─────────────────────────────────────────────────┘
         │
         ▼
  ┌─────────────┐
  │ domhand_fill │ ◀── uses observation to fill fields
  └─────────────┘
         │
         ▼
  ┌─────────────┐
  │Fill Executor │ ◀── unchanged, stays as-is
  └─────────────┘
```

### Layer 1: Raw Extractor — "What's physically on the page"

- Single `page.evaluate()` call extracts all interactive elements
- Returns raw facts: tag, role, attributes, bounding rect, visibility, DOM path
- **No interpretation** — just data collection
- Can parallelize CDP calls (DOM snapshot + a11y tree + metadata)

### Layer 2: Semantic Interpreter — "What each element means"

- **Runs in Python, not JavaScript** — this is the core improvement
- Sub-components:
  - **Type Classifier** — maps raw role/tag/attributes → canonical types (text, select, radio_group, checkbox_group, date, file, toggle, button_group)
  - **Label Resolver** — resolves accessible name with confidence scoring
  - **Field Grouper** — associates questions with their options (handles the sibling problem)
  - **Option Discoverer** — extracts dropdown options from native selects, ARIA listboxes, portal-based dropdowns
- Produces `SemanticField` objects (successor to `FormField`)

### Layer 3: State Tracker — "What changed since last observation"

- Maintains field state across agent steps
- Detects: newly revealed fields, value changes, page transitions, validation errors
- Uses MutationObserver for dirty-field detection (not always-on — attribute-filtered, debounced)
- Replaces hardcoded `asyncio.sleep()` with proper DOM stability waiting
- Produces observation diffs for LLM context compression

### Key Design Decisions

| Decision | Rationale |
|----------|-----------|
| **Pipeline, not event-driven** | Layers have strict data dependencies (L2 needs L1, L3 needs L2). Observation happens at discrete points in the agent loop, not continuously. |
| **Python over JavaScript for interpretation** | Current 500+ line JS blob can't be unit tested or debugged properly. Python gives us logging, error handling, testability. |
| **Strategy pattern for extraction** | Swap extraction backends (DOM-only → hybrid DOM+a11y) without changing interpreter or state tracker |
| **Adapter pattern for migration** | `SemanticField` initially mirrors `FormField` interface so fill executor doesn't need to change immediately |
| **LLM-free observation** | Observation pipeline is 100% deterministic. LLM only enters at two points: agent planner (browser-use) and fill answer generator (inside domhand_fill) |
| **Build on browser-use, not parallel to it** | browser-use already fetches DOM + a11y tree + snapshot in parallel via `DomService`. We layer on top, avoiding duplicate CDP calls |

## 3. Observation Strategies Analyzed

We evaluated four approaches:

| Strategy | Determinism | Cost | Latency | Widget Support | Verdict |
|----------|------------|------|---------|----------------|---------|
| **A: Pure A11y Tree** | HIGH | $0 | ~50ms | MEDIUM — gaps for custom widgets | Not viable alone (can't target elements for actions) |
| **B: DOM + A11y Tree (hybrid)** | HIGH | $0 | ~80ms | HIGH — DOM fills a11y gaps | **RECOMMENDED** — best of both worlds |
| **C: A11y + Vision** | LOW | $0.01-0.03/page | ~600ms | HIGH | Valid as escalation only, not primary |
| **D: Vision-First** | LOW | $0.01-0.03/page | ~600ms | HIGH | REJECTED — too expensive, non-deterministic |

**Winner: Strategy B (Hybrid DOM + A11y Tree)**

- DOM provides stable element references for fill actions
- A11y tree provides computed accessible names, roles, and states
- browser-use already does this merge (`EnhancedDOMTreeNode`)
- Graceful degradation: if a11y is missing info, fall back to DOM-only heuristics

## 4. What the A11y Tree Gives Us

The Chrome Accessibility Tree (`CDP Accessibility.getFullAXTree`) is the industry consensus primary signal for web agent observation. Here's what it provides vs raw DOM:

| Signal | Raw DOM | A11y Tree | Winner |
|--------|---------|-----------|--------|
| Field semantics (role) | Must infer from tag + attributes | Explicit `role` property | **A11y tree** |
| Accessible name (label) | Complex resolution chain | Pre-computed `name` property | **A11y tree** |
| State (checked, expanded, selected) | Scattered across attributes | Unified `properties` array | **A11y tree** |
| Shadow DOM content | Requires cross-shadow traversal | Flattened into single tree | **A11y tree** |
| Element targeting for actions | Direct via selectors | Only via cross-reference | **Raw DOM** |
| Custom widget internals | Opaque without widget-specific logic | Only what ARIA exposes | **Depends on ARIA quality** |

**Industry consensus** (WebArena, Stagehand, rtrvr.ai, browser-use): a11y tree as primary semantic signal, DOM for structural anchoring, vision only as escalation.

## 5. Cost Analysis

| Scenario | Observation Cost | Notes |
|----------|-----------------|-------|
| DOM + A11y extraction (per page) | **$0** | Pure CDP calls, no LLM |
| 5-page standard application | **~$0.0004** | With strategic screenshots via Gemini Flash |
| 8-page Workday application | **~$0.0006** | With strategic screenshots |
| Current fill LLM cost (for comparison) | **$0.01-0.05** | Fill LLM dominates total cost |

Observation is essentially free. The fill LLM call is 25-125x more expensive.

**Per-action screenshots would cost $0.75-7.20 per application** — this is why we rejected Strategy D.

## 6. Critical Risks (Top 5)

### C1: Breaking the domhand_fill Contract

`fill_executor.py` depends on `FormField`'s exact field names, `data-ff-id` tagging, and `field_type` string values. All 20+ browser scripts use these.

**Prevention:** New model is strict superset of FormField. Adapter bridges during migration. Run both extractors in parallel shadow mode. Never change field_type enum values.

### C2: A11y Tree Gaps for Custom Widgets

Button groups have no ARIA role. Workday's `data-uxi-widget-type` isn't standard. Cross-shadow-root `aria-labelledby` breaks.

**Prevention:** Spike on 3 real ATS platforms in Phase 1 before committing. Plan hybrid approach from day one. Keep the `_JS_DETECT_BUTTON_GROUPS` pass.

### C3: Regression on Working Platforms

6 ATS platforms (Workday, Greenhouse, Lever, Oracle, SmartRecruiters, Phenom) each have unique widget patterns. Can't ship "better for Workday but broken for Greenhouse."

**Prevention:** Platform parity matrix with HTML fixtures. Gate shipping on all-platform parity. Platform-by-platform rollout behind feature flag.

### C4: getFullAXTree Performance on Workday

5000+ DOM nodes across shadow roots. CDP call could take 2-8 seconds.

**Prevention:** Benchmark on real Workday pages in Phase 1 spike. Use depth-limited fetch. Set 2-second observation budget with fallback.

### H5: LLM Context Overflow

The #1 documented problem. Too many fields + too many options → hallucinations.

**Prevention:** Section-scoped field context. Prune dropdown options to top 10 fuzzy matches. 2000-token budget per page observation. Observation diffs (only send what changed).

## 7. What Varies Across ATS Platforms

| Dimension | Workday | Greenhouse/Lever | Oracle HCM | SmartRecruiters | Phenom |
|-----------|---------|-------------------|------------|-----------------|--------|
| **Widget system** | Custom `data-uxi-widget-type` | React components, standard HTML | JET custom elements (`oj-*`) | Standard HTML + React | React + custom |
| **Select impl** | Portal + virtualized lists | React Select, native `<select>` | cx-select custom elements | Standard `<select>` | React Select |
| **Shadow DOM** | Yes (inner widgets) | Minimal | Yes (JET) | Minimal | Minimal |
| **Label mechanism** | `data-automation-id="fieldLabel"` | `label[for]` | `aria-labelledby` | `label[for]`, ARIA | `label[for]`, ARIA |
| **Date input** | Segmented (MM/DD/YYYY separate) | Single date picker | Segmented or calendar | Single date picker | Single date picker |
| **Repeater** | Numbered headings | Saved tile badges | Section "Add" buttons | Similar to GH | Similar to GH |

**What's consistent** (build generic logic around):
- ARIA roles for interactive elements ✓
- `aria-required="true"` for required fields ✓
- Heading elements (h1-h4) for sections ✓
- `role="option"` inside `role="listbox"` for dropdowns ✓
- CSS `display:none` / `visibility:hidden` for hidden elements ✓

## 8. Features Breakdown

### Must Ship (v2.0)

1. **Deterministic field discovery with stable ordering** — same page = same output, always
2. **Improved label resolution with confidence tracking** — including sibling-based labels
3. **Canonical widget taxonomy** — declarative pattern registry replacing scattered type resolution
4. **Structured observation with diff support** — "what changed" vs full dump for LLM context
5. **DOM stability waiting** — MutationObserver-based settle detection replacing hardcoded sleeps
6. **Loading state detection** — detect spinners/skeleton screens before observing

### Should Ship (v2.0 if time allows)

- Observation diff for LLM context compression
- Field completeness scoring per page ("17/20 filled, 2 required empty")
- Confidence scoring per field
- Semantic section classification (heading variations → canonical categories)
- Platform detection heuristic with widget fingerprint library

### Defer to v2.1+

- Generic repeater section detection (current platform-specific approach works)
- Nested group hierarchy (current flat list is functional)
- Question intent detection (LLM-dependent, layer on after core is solid)

### Anti-Features (Deliberately Avoid)

| Don't Do This | Why |
|---------------|-----|
| Per-action screenshots | Cost spiral: $0.75-7.20 per application |
| Full DOM serialization to LLM | Documented cause of hallucinations |
| Platform-specific extraction code paths | Use declarative pattern registry instead |
| LLM-based field type classification | Must be deterministic and instant |
| Always-on MutationObserver | Performance killer on Workday/Oracle |

## 9. Proposed Build Order (6 Phases)

```
Phase 1 ──▶ Phase 2 ──▶ Phase 3 ──▶ Phase 4 ──▶ Phase 5 ──▶ Phase 6
Contract    Raw         Semantic     State +      Platform    A11y Tree
+ Spike     Extractor   Interpreter  Integration  Parity      Enrichment
+ Fixtures  (Layer 1)   (Layer 2)    (Layer 3)    + Rollout   (Strategy B)
[LOW RISK]  [LOW RISK]  [MED RISK]   [MED RISK]   [MED RISK]  [HIGH RISK]
```

### Phase 1: Contract + Spike + Test Fixtures

**Goal:** Validate the approach before writing production code.

**Delivers:**
- Data model definitions (Observation, RawPageSnapshot, SemanticField, ObservationState)
- Adapter from SemanticField → FormField for backward compatibility
- HTML snapshot fixtures from 6 ATS platforms (2+ pages each)
- Golden-file expected outputs from current extractor
- CDP a11y tree spike: run `getFullAXTree` on real Workday/Oracle/Greenhouse pages, document every gap
- Performance benchmarks for `getFullAXTree` on Workday

**Why first:** You cannot build on an unvalidated foundation. The spike answers two existential questions: (1) is the contract compatible? (2) does the a11y tree actually work on our target platforms?

### Phase 2: Raw Extractor (Layer 1)

**Goal:** Wrap existing JS extraction in the new interface.

**Delivers:**
- `RawExtractor` class producing `RawPageSnapshot`
- Simplified JS that extracts raw facts only (no type classification, no label resolution, no grouping)
- Parity tests: old output matches new output on all fixtures

**Why second:** Foundation for the pipeline. Low risk — structural refactor of existing code.

### Phase 3: Semantic Interpreter (Layer 2)

**Goal:** Port interpretation logic from JS to Python. This is the core value of the rebuild.

**Delivers:**
- Type Classifier (canonical widget taxonomy)
- Label Resolver (confidence scoring + sibling heuristics)
- Field Grouper (radio/checkbox groups, button groups, date groups)
- Option Discoverer
- Section-scoped field partitioning

**Why third:** This is where the most value lives. Moving opaque JS to testable Python. Depends on Layer 1.

### Phase 4: State Tracker (Layer 3) + domhand_fill Integration

**Goal:** Close the observe-act-verify loop.

**Delivers:**
- StateTracker with field-level diff detection
- MutationObserver dirty-field tracking (attribute-filtered, debounced)
- DOM stability waiting (replacing hardcoded asyncio.sleep)
- Feature-flagged integration: `GH_USE_NEW_OBSERVER=1`
- Observation diff for LLM context compression

**Why fourth:** Depends on stable SemanticField output from Phase 3.

### Phase 5: Platform Parity + Rollout

**Goal:** All 6 ATS platforms must pass before replacing the old extractor.

**Delivers:**
- Platform parity matrix — all 6 passing golden-file tests
- Platform-by-platform feature flag rollout
- Widget fingerprint library (declarative pattern registry)
- iframe content extraction for Greenhouse/Oracle/Phenom

**Why fifth:** The gating phase. No platform-by-platform rollout until all pass regression.

### Phase 6: A11y Tree Enrichment (Strategy B)

**Goal:** Swap to hybrid DOM+a11y extraction strategy.

**Delivers:**
- `HybridExtractionStrategy` using browser-use's `EnhancedDOMTreeNode`
- A11y-enriched label resolution (browser-computed accessible names)
- A11y-enriched state detection (checked, expanded, selected from a11y properties)
- Side-by-side comparison on all fixtures

**Why last:** Highest impact, highest risk. Strategy pattern means it can be added without touching interpreter or state tracker.

## 10. The Observer Contract (OOD)

### Data Flow

```
Page Load / SPA Navigation
         │
         ▼
[Raw Extractor]  ──────▶  RawPageSnapshot
         │                 (elements, metadata, fingerprint)
         ▼
[Semantic Interpreter]  ──▶  SemanticPageModel
         │                   (typed fields, sections, groups)
         ▼
[State Tracker]  ──────▶  ObservationState
         │                 (diffs, reveals, errors)
         ▼
    Observation
    ┌──────────────┐
    │ raw:          │ ── Layer 1 output
    │ semantic:     │ ── Layer 2 output
    │ state:        │ ── Layer 3 output
    │ metadata:     │ ── timing, strategy used
    └──────────────┘
         │
    ┌────┴────┐
    │         │
    ▼         ▼
domhand_fill  domhand_assess_state
    │              │
    ▼              ▼
Fill LLM Call  State Classification
(Haiku/Flash)  (pure Python)
    │
    ▼
Fill Executor ──▶ ActionFeedback ──▶ back to State Tracker
```

### Key Interfaces

```python
# Observer — the main entry point
class Observer:
    async def observe(page, context?) -> Observation

# Extraction strategies — swappable backends
class ExtractionStrategy(Protocol):
    async def extract(page) -> RawPageSnapshot

# State change callbacks — for the agent loop
class StateChangeCallback(Protocol):
    def on_fields_revealed(fields) -> None
    def on_validation_errors(errors) -> None
    def on_page_transition(from_page, to_page) -> None

# Action feedback — closing the loop
class ActionFeedback:
    field_results: list[FillFieldResult]
    fields_attempted, succeeded, failed: list[str]
    page_navigated: bool
```

### Boundary Rules

- Observer **NEVER** fills fields or clicks elements
- Fill Executor **NEVER** extracts or interprets fields
- DomHand Actions are the **only bridge** between observation and action
- browser-use Agent sees **tool results**, never raw observation data
- State Tracker is the **only** component with cross-step memory

## 11. Open Questions for Team Discussion

1. **A11y tree viability per platform** — How complete is Workday's a11y tree for their custom widgets? Oracle? Greenhouse? This is the single biggest unknown. Phase 1 spike will answer this.

2. **getFullAXTree performance** — Is it fast enough on Workday's 5000+ node pages? We need real benchmarks before committing.

3. **iframe handling** — Greenhouse/Oracle iframes: cross-origin or same-origin? browser-use has `max_iframes` but we haven't tested it systematically.

4. **Virtualized dropdown lists** — React Select only renders visible options. Does the a11y tree see all options or only rendered ones?

5. **Repeater detection scope** — Should we try generic repeater detection in v2.0 or keep the current platform-specific approach and defer to v2.1?

6. **Migration strategy** — Feature flag per platform? Or parallel shadow mode comparing old vs new? Both? What's the rollout comfort level?
