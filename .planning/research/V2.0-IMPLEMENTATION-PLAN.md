## System Overview When Complete

When all six phases are done, observation works like this:

1. The **Observer** (session-scoped, created once per job) receives a Playwright `Page`.
2. **Layer 1 (RawExtractor)** calls browser-use's `DomService.get_dom_tree()` to get merged DOM + a11y data via CDP, tags interactive elements with `data-ff-id`, and produces a `RawPageSnapshot` containing `RawElement` objects with both DOM and a11y signals.
3. **Layer 2 (SemanticInterpreter)** runs five deterministic sub-components (TypeClassifier, LabelResolver, SectionClassifier, OptionDiscoverer, ConfidenceScorer) plus one async LLM call (FieldGrouper) to produce a `SemanticPageModel` of fully interpreted `SemanticField` objects.
4. **Layer 3 (StateTracker)** diffs the current `SemanticPageModel` against the previous observation, using a JS MutationObserver for dirty-element tracking, and produces an `ObservationDiff`.
5. The **FormFieldAdapter** converts `SemanticField` back to `FormField` / `ExtractionResult`, making the new pipeline a drop-in replacement for `extract_form_fields()`. The entire action layer (domhand_fill, fill_executor, verification_engine, 20+ browser scripts) remains untouched.

The old `field_extractor.py` (866-line JS blob that does extraction + interpretation in one pass) is replaced. The new system is testable without a browser (Layers 2-3 are pure Python), extensible per platform (no hardcoded checks in JS), and produces confidence metadata for every classification decision.

## Phase 1: Layer 1 — RawExtractor + A11y Tree Integration

### Goal

Build the raw data collection layer that replaces the JS extraction blob with browser-use's `DomService`. Produce `RawElement` and `RawPageSnapshot` models populated with both DOM and a11y tree signals. Answer the four open questions about a11y tree quality (OOD Section 13, questions 1-4).

### Dependencies

None. This is the foundation phase.

### Deliverables

| File | What it contains |
|------|-----------------|
| `ghosthands/dom/observation/models.py` | `RawElement`, `RawPageSnapshot` Pydantic models (OOD Section 4.1-4.2) |
| `ghosthands/dom/observation/raw_extractor.py` | `RawExtractor` class: calls `DomService.get_dom_tree()`, walks `EnhancedDOMTreeNode`, maps to `RawElement`, handles ff_id tagging (OOD Section 5) |
| `ghosthands/dom/observation/__init__.py` | Package init with public exports |
| `tests/unit/observation/test_raw_element.py` | Unit tests for `RawElement` model validation and edge cases |
| `tests/unit/observation/test_raw_extractor.py` | Unit tests for `_walk_tree()` logic using mock `EnhancedDOMTreeNode` fixtures |
| `tests/integration/observation/test_raw_extractor_live.py` | Integration tests that run `RawExtractor.extract()` against real ATS pages (Workday, Greenhouse, Lever) to answer open questions 1-4 |
| `scripts/benchmark_a11y_quality.py` | Benchmark script that extracts from real pages and reports: ax_role coverage %, ax_name coverage %, `getFullAXTree` latency, virtualized dropdown visibility, Workday label resolution quality |

### Acceptance Criteria

- [ ] `RawExtractor.extract(page)` returns a `RawPageSnapshot` with all interactive elements found on a test page
- [ ] Every `RawElement` has `backend_node_id` and `ff_id` populated
- [ ] Elements in shadow DOM and cross-origin iframes are included (browser-use handles this via `cross_origin_iframes=True`)
- [ ] `ax_role`, `ax_name`, and `ax_properties` are populated for elements where the a11y tree provides data
- [ ] `data-ff-id` attributes are correctly tagged on DOM elements (verified by `page.querySelector('[data-ff-id="ff-0"]')` returning a valid element)
- [ ] `_walk_tree()` can be tested with mock tree fixtures (no browser needed)
- [ ] Benchmark results for open questions 1-4 are documented with actual numbers from real ATS pages

### Open Questions Answered

| # | Question | How answered |
|---|----------|-------------|
| 1 | What % of fields have correct `ax_role`? | `benchmark_a11y_quality.py` reports per-platform coverage |
| 2 | `getFullAXTree` latency on large pages? | Benchmark measures wall-clock time on Workday pages with 1000+ nodes |
| 3 | A11y tree + virtualized dropdowns? | Integration test opens a Workday dropdown with 100+ options, checks if a11y tree sees them all |
| 4 | `ax_name` on Workday `data-automation-id` labels? | Integration test compares `ax_name` to expected label on real Workday fields |

### Risks

| Risk | Impact | Mitigation |
|------|--------|------------|
| `DomService.get_dom_tree()` API changes in upstream browser-use | Breaks the extraction layer | browser-use is vendored in-repo; pin the interface and add a compatibility shim if needed |
| `getFullAXTree` is too slow on large pages (>2s) | Observation exceeds time budget | Benchmark early (this is why Phase 1 includes the benchmark). If slow, investigate depth-limiting or selective a11y enrichment for only interactive elements |
| A11y tree coverage is low (<70% of fields have useful `ax_role`) | Reduces the value of the a11y integration | The system still works with DOM-only signals; TypeClassifier (Phase 2) uses multi-signal scoring with fallback weights. Low a11y coverage just means lower confidence scores, not failures |
| ff_id tagging race condition with dynamic pages | Elements tagged before page finishes rendering get stale IDs | Use `DomService.get_dom_tree()` which waits for DOM stability before snapshot. Add a stability check before tagging. |

## Phase 2: Layer 2 Deterministic Components

### Goal

Build the five deterministic sub-components of the Semantic Interpreter: TypeClassifier, LabelResolver, SectionClassifier, OptionDiscoverer, and ConfidenceScorer. These are all pure Python functions that take `RawElement` inputs and produce classification outputs with no browser or LLM dependency.

### Dependencies

- **Phase 1** must be complete (provides `RawElement` and `RawPageSnapshot` models).

### Deliverables

| File | What it contains |

| `ghosthands/dom/observation/semantic/label_resolver.py` | `LabelResolver` with priority-ordered source chain (OOD Section 6.2). Maps `RawElement` to `(label, LabelSource, confidence)` |
| `ghosthands/dom/observation/semantic/section_classifier.py` | `SectionClassifier` with canonical section name mapping (OOD Section 6.5). Maps heading text to canonical names |
| `ghosthands/dom/observation/semantic/option_discoverer.py` | `OptionDiscoverer` that enumerates choices for select/radio/checkbox fields (OOD Section 6.4). Returns `list[FieldOption]` |
| `ghosthands/dom/observation/semantic/confidence_scorer.py` | `ConfidenceScorer` that produces `FieldConfidence` from type + label classification results |
| `ghosthands/dom/observation/semantic/models.py` | `CanonicalType`, `LabelSource`, `FieldConfidence`, `SemanticField`, `SemanticSection`, `SemanticPageModel` Pydantic models (OOD Section 4.3-4.4) |
| `ghosthands/dom/observation/semantic/__init__.py` | Package init |
| `tests/unit/observation/semantic/test_type_classifier.py` | Exhaustive tests: every CanonicalType, multi-signal agreement, signal disagreement, platform-specific widgets (Workday `data-uxi-widget-type`, etc.) |
| `tests/unit/observation/semantic/test_label_resolver.py` | Tests: priority order, ax_name used when available, fallback chain, Workday-specific label patterns |
| `tests/unit/observation/semantic/test_section_classifier.py` | Tests: canonical mapping, case insensitivity, unknown headings |
| `tests/unit/observation/semantic/test_option_discoverer.py` | Tests: native select, ARIA listbox, radio labels, checkbox labels |
| `tests/unit/observation/semantic/test_confidence_scorer.py` | Tests: high confidence when signals agree, low confidence on disagreement, warning generation |

### Acceptance Criteria

- [ ] `TypeClassifier.classify(raw)` is a pure function testable with dictionary-constructed `RawElement` objects (no browser, no page.evaluate)
- [ ] TypeClassifier correctly handles all 8 `CanonicalType` values with test cases for each
- [ ] TypeClassifier uses multi-signal scoring: when `ax_role` agrees with `tag_name + input_type`, confidence is > 0.9. When they disagree, confidence drops and a warning is emitted
- [ ] LabelResolver returns `ax_computed_name` as the label when available, falling back through the priority chain (OOD Section 6.2 `PRIORITY_ORDER`)
- [ ] SectionClassifier maps common variations ("Work Experience", "Employment History", "Work History") to a single canonical name
- [ ] OptionDiscoverer handles native `<select>` elements, ARIA listbox patterns, and radio/checkbox labels
- [ ] ConfidenceScorer produces `FieldConfidence` with `type_signals` listing which signals contributed
- [ ] All tests pass with `pytest tests/unit/observation/semantic/ -v` and require zero external dependencies

### Open Questions Answered

None directly. Phase 2 builds on the a11y quality data gathered in Phase 1 to calibrate signal weights in TypeClassifier.

### Risks

| Risk | Impact | Mitigation |
|------|--------|------------|
| Type classification accuracy is lower than the old JS extractor for certain platforms | Regression in field detection | Build test fixtures from real page extractions (Phase 1 benchmark data). Compare classifier output to the old extractor's decisions field-by-field. Adjust signal weights until parity is reached. |
| Label priority order is wrong for some ATS platforms | Fields get incorrect labels | LabelResolver logs which source it chose. Integration tests in Phase 5 will catch label regressions. The priority order can be tuned per platform if needed. |
| Too many CanonicalType edge cases not covered | Misclassified fields | Start with the 8 types from the OOD. If real data reveals a gap, add sub-types or platform hints. The `warnings` field on SemanticField flags uncertain classifications. |

## Phase 3: Layer 2 LLM Grouper

### Goal

Build the LLM-assisted Field Grouper with a provider-agnostic interface, at least two provider implementations, a heuristic fallback, and the structured prompt. Answer the four open questions about LLM grouping quality (OOD Section 13, questions 7-10).

### Dependencies

- **Phase 1** must be complete (provides `RawElement` for building `GroupingCandidate` objects).
- **Phase 2** is NOT required. The Grouper takes pre-classified type/label/section data as inputs; for testing, these can be provided directly without the real classifiers.

### Deliverables

| File | What it contains |
|------|-----------------|

| `ghosthands/dom/observation/grouping/protocol.py` | `GroupingLLMProvider` protocol definition (OOD Section 6.3) |
| `ghosthands/dom/observation/grouping/siliconflow_provider.py` | `SiliconFlowProvider` implementation |

| `ghosthands/dom/observation/grouping/field_grouper.py` | `FieldGrouper` class with LLM call, 2-second timeout, and `_heuristic_fallback()` (OOD Section 6.3) |
| `ghosthands/dom/observation/grouping/prompt.py` | Grouping prompt template (OOD Section 6.3 prompt specification) |
| `ghosthands/dom/observation/grouping/__init__.py` | Package init |
| `tests/unit/observation/grouping/test_field_grouper.py` | Tests: mock LLM provider, heuristic fallback on timeout, heuristic fallback on malformed response, correct candidate filtering |
| `tests/unit/observation/grouping/test_heuristic_fallback.py` | Tests: same-label grouping, same-section grouping, edge cases (single element groups, empty input) |
| `tests/integration/observation/grouping/test_grouper_providers.py` | Integration tests: real LLM calls to each provider with known-correct grouping fixtures. Measures latency and accuracy. |
| `scripts/benchmark_grouping_providers.py` | Benchmark script: runs the same set of real page snapshots through each provider, reports accuracy, latency (p50/p95), cost, structured output reliability |

### Acceptance Criteria

- [ ] `FieldGrouper.group()` calls the LLM provider and returns a valid `GroupingResponse`
- [ ] If the LLM call exceeds 2 seconds, `asyncio.TimeoutError` is caught and `_heuristic_fallback()` is used
- [ ] If the LLM returns malformed JSON, fallback is used (no crash)
- [ ] `_heuristic_fallback()` produces same-label + same-section grouping (parity with current `_group_fields()` behavior)
- [ ] `GroupingLLMProvider` is a `Protocol` class -- any object with `async def group_fields(candidates, page_context) -> GroupingResponse` satisfies it
- [ ] Both `SiliconFlowProvider` and `OpenAIProvider` pass integration tests with real API calls
- [ ] Benchmark results for open questions 7-10 are documented with actual numbers

### Open Questions Answered

| # | Question | How answered |
|---|----------|-------------|
| 7 | Best cheap LLM provider for grouping? | Benchmark compares accuracy across providers on real snapshots |
| 8 | p95 latency of grouping LLM call? | Benchmark measures p50/p95 latency per provider |
| 9 | Structured output reliability from cheap models? | Benchmark tracks JSON parse failure rate per provider |
| 10 | Can grouping prompts be cached/templated per platform? | Analyze prompt similarity across same-platform pages in benchmark data |

### Risks

| Risk | Impact | Mitigation |
|------|--------|------------|
| No cheap model produces reliable structured output | Grouping falls back to heuristics on most calls, losing the key improvement | Test multiple providers (Silicon Flow, GPT-5.2 Nano, Gemini Flash). If all fail, implement a retry-with-simpler-prompt strategy or switch to a slightly more expensive model that supports guaranteed JSON mode. |
| Grouping latency exceeds 500ms at p95 | Observation time budget (~2s total) gets squeezed | The 2-second hard timeout ensures we never block. If latency is consistently high, reduce prompt size by sending fewer candidates (only groupable elements, not all elements). |
| LLM grouping accuracy is not better than heuristic fallback | No improvement over current system, but added cost and complexity | If benchmark shows <80% accuracy improvement over heuristics, reconsider the LLM approach. The heuristic fallback ensures we never regress below current quality. |
| Rate limiting from LLM provider during high-volume runs | Grouping falls back to heuristics frequently | Implement provider rotation (try secondary provider if primary is rate-limited). The fallback path is always available. |

## Phase 4: Layer 3 StateTracker + MutationObserver + Adapter

### Goal

Build the StateTracker (Layer 3) with MutationObserver-based dirty-element tracking, the FormFieldAdapter for backward compatibility, and the top-level `Observer` class that wires all three layers together. After this phase, the full pipeline exists and can be called end-to-end.

### Dependencies

- **Phase 1** must be complete (Layer 1 extraction).
- **Phase 2** must be complete (deterministic Layer 2 components).
- **Phase 3** must be complete (LLM Grouper).

### Deliverables

| File | What it contains |
|------|-----------------|
| `ghosthands/dom/observation/state/models.py` | `FieldChange`, `ObservationDiff`, `ObservationState` Pydantic models (OOD Section 4.5) |
| `ghosthands/dom/observation/state/state_tracker.py` | `StateTracker` class: `diff()`, `get_dirty_elements()`, `wait_for_stable()` (OOD Section 7) |
| `ghosthands/dom/observation/state/mutation_observer.js` | JS MutationObserver script: attribute-filtered, debounced, reports dirty ff_ids. Activated between fill steps, not always-on (OOD decision D5). |
| `ghosthands/dom/observation/adapter.py` | `FormFieldAdapter`: `to_form_field()`, `to_extraction_result()` (OOD Section 8). Maps SemanticField to FormField with exact field compatibility. |
| `ghosthands/dom/observation/observer.py` | `Observer` class: session-scoped singleton, wires Layer 1 + 2 + 3 + Adapter (OOD Section 10.2). Provides `observe()` (returns `ExtractionResult`) and `observe_semantic()` (returns `ObservationDiff`). |
| `ghosthands/dom/observation/semantic/interpreter.py` | `SemanticInterpreter` class: orchestrates TypeClassifier + LabelResolver + SectionClassifier + FieldGrouper + OptionDiscoverer + ConfidenceScorer (OOD Section 6.6) |

| `tests/unit/observation/test_observer.py` | Tests: end-to-end with mock DomService and mock LLM provider, verify `observe()` returns valid `ExtractionResult` |
| `tests/integration/observation/test_observer_e2e.py` | Integration test: run Observer against a real page, verify output is structurally valid |

### Acceptance Criteria

- [ ] `Observer.observe(page)` returns an `ExtractionResult` with the same schema as the current `extract_form_fields()` output
- [ ] `FormFieldAdapter.to_form_field()` produces output that passes the existing `FormField` Pydantic validation
- [ ] Every `CanonicalType` maps to the correct old `field_type` string (OOD Section 8 TYPE_MAP)
- [ ] `index` defaults to `-1`, `selector` defaults to `""`, `name`/`placeholder` default to `""` (OOD Section 8.1)
- [ ] `StateTracker.diff()` returns `is_new_page=True` on the first call and on URL changes
- [ ] `StateTracker.diff()` correctly detects field additions, removals, and value changes on subsequent calls
- [ ] `wait_for_stable()` uses MutationObserver (not `asyncio.sleep()`) to detect DOM stability
- [ ] `Observer` is session-scoped: `StateTracker` preserves state across multiple `observe()` calls within the same instance (OOD decision D3)
- [ ] Full pipeline (extract -> interpret -> diff -> adapt) completes within 3 seconds on a typical application page

### Open Questions Answered

None. All open questions are addressed in Phases 1 and 3.

### Risks

| Risk | Impact | Mitigation |
|------|--------|------------|
| Adapter produces FormField that is subtly different from old extractor output | Action layer behaves differently, fills fail | Phase 5 (shadow mode) catches this. Build adapter tests from real old-extractor output fixtures: serialize actual `extract_form_fields()` output, then verify the adapter produces identical output for the same page. |
| MutationObserver misses changes in shadow DOM | StateTracker doesn't detect field mutations inside shadow roots | Test MutationObserver on shadow-DOM-heavy platforms (Workday, Oracle). If shadow DOM mutations are missed, add per-shadow-root observers. |
| SemanticInterpreter orchestration order matters (e.g., grouper needs types first) | Incorrect pipeline results | The orchestration order is specified in OOD Section 6.6: types -> labels -> sections -> grouping -> options -> confidence -> merge. Follow this exactly. Unit test the full interpreter with known inputs and expected outputs. |

## Phase 5: Shadow-Mode Parity Testing

### Goal

Run the new Observer in parallel with the old extractor on real production traffic. Compare outputs field-by-field. Identify and fix all discrepancies until parity is achieved. The old extractor remains the source of truth throughout this phase -- the new observer's output is logged but never used for actual fills.

### Dependencies

- **Phase 4** must be complete (full pipeline including adapter exists).

### Deliverables

| File | What it contains |
|------|-----------------|
| `ghosthands/dom/observation/shadow_mode.py` | `extract_with_shadow_validation()`: runs both extractors, compares, logs mismatches (OOD Section 11) |
| `ghosthands/dom/observation/comparison.py` | `compare_extraction_results()`: field-by-field diff between old `ExtractionResult` and new `ExtractionResult`. Reports mismatches by field_id, field_type, label, options, value, required, visible. |
| `scripts/analyze_shadow_mismatches.py` | Analysis script: reads shadow-mode logs, aggregates mismatch statistics by platform, field type, and mismatch category |
| Updated `ghosthands/actions/domhand_fill.py` | Minimal change: accept optional `observer: Observer` parameter in `extract_visible_form_fields()` (OOD Section 10.3) |
| Updated `ghosthands/config/settings.py` | Add `GH_USE_NEW_OBSERVER` and `GH_GROUPING_PROVIDER` env var support |

### Acceptance Criteria

- [ ] Shadow mode can be activated with `GH_USE_NEW_OBSERVER=shadow` (runs both, returns old result)
- [ ] Every mismatch is logged as a structured log event with: platform, page URL, field_id, mismatch type, old value, new value
- [ ] Mismatch analysis script can produce a per-platform report showing: total fields compared, match rate, top mismatch categories
- [ ] After iterating on fixes: **95% field-level match rate** across all platforms combined
- [ ] Per-platform match rates: Lever >= 98%, Greenhouse >= 97%, SmartRecruiters >= 95%, Phenom >= 93%, Oracle >= 90%, Workday >= 90%
- [ ] No new observer crash/exception on any real page (graceful degradation to old extractor if observer fails)
- [ ] Total observation time (new pipeline) is within 1.5x of the old extractor's time on the same page

### Open Questions Answered

None formally, but shadow mode will surface any remaining edge cases not covered by the OOD.

### Risks

| Risk | Impact | Mitigation |
|------|--------|------------|
| Running both extractors doubles observation time | Fill loops slow down for all users during shadow mode | Shadow mode runs the new observer in a background task (`asyncio.create_task`). The old extractor returns immediately; the comparison happens asynchronously. If the new observer is too slow, it times out without blocking the fill. |
| Mismatch volume is too high to fix one-by-one | Phase drags on with diminishing returns | Categorize mismatches and fix the highest-impact categories first. Accept that some edge cases (e.g., a rare Oracle widget) may have lower match rates at cutover, with platform-specific tuning done post-cutover. |
| Shadow mode introduces a subtle bug in the old path | Production regression during testing phase | The old extraction path is completely unchanged. `extract_visible_form_fields()` only adds an `if observer is not None` branch that runs the new path. The old `else` branch is identical to the current code. |

## Phase 6: Platform-by-Platform Cutover

### Goal

Replace the old extractor with the new Observer, one ATS platform at a time, starting with the simplest platforms and ending with the most complex. Each platform is gated by an independent feature flag. The old extractor is removed only after all platforms have been cut over and validated.

### Dependencies

- **Phase 5** must be complete (shadow-mode parity validated with >= 95% overall match rate).

### Deliverables

| File | What it contains |
|------|-----------------|
| Updated `ghosthands/config/settings.py` | Per-platform feature flags: `GH_NEW_OBSERVER_LEVER`, `GH_NEW_OBSERVER_GREENHOUSE`, etc. |
| Updated `ghosthands/actions/domhand_fill.py` | Platform-aware observer routing: check platform from URL, check per-platform flag, route to new or old extractor |
| Updated `ghosthands/dom/observation/observer.py` | Add platform detection utility (used by feature flag routing) |
| `scripts/cutover_validation.py` | Per-platform validation script: runs N real applications through the new observer, verifies fill success rate matches or exceeds the old extractor |
| Monitoring dashboards / alerts | Structured log queries for: observation errors by platform, fill success rate by platform, observation latency by platform |

### Cutover Order

Following the OOD's suggested rollout order (OOD Section 11):

| Order | Platform | Why this position | Flag |
|-------|----------|-------------------|------|
| 1 | Lever | Simplest ATS, mostly native HTML elements | `GH_NEW_OBSERVER_LEVER=1` |
| 2 | Greenhouse | Standard ARIA patterns, well-structured | `GH_NEW_OBSERVER_GREENHOUSE=1` |
| 3 | SmartRecruiters | Moderate complexity | `GH_NEW_OBSERVER_SMARTRECRUITERS=1` |
| 4 | Phenom | Moderate complexity with some iframes | `GH_NEW_OBSERVER_PHENOM=1` |
| 5 | Oracle | Complex custom widgets | `GH_NEW_OBSERVER_ORACLE=1` |
| 6 | Workday | Most complex, most custom, highest stakes | `GH_NEW_OBSERVER_WORKDAY=1` |

### Acceptance Criteria (per platform)

- [ ] Feature flag is off by default; enabled only after validation
- [ ] At least 20 real applications run successfully through the new observer on that platform
- [ ] Fill success rate is equal to or better than the old extractor on that platform
- [ ] No increase in observation-related error rate
- [ ] Observation latency does not regress by more than 500ms compared to the old extractor
- [ ] Flag can be instantly rolled back to the old extractor if issues arise

### Final cleanup (after all platforms cut over)

- [ ] Remove `field_extractor.py` (the 866-line JS extraction blob)
- [ ] Remove shadow-mode comparison code
- [ ] Remove per-platform feature flags (set new observer as the only path)
- [ ] Remove `FormFieldAdapter` only when the action layer is updated to consume `SemanticField` directly (future milestone, not part of this plan)

### Risks

| Risk | Impact | Mitigation |
|------|--------|------------|
| A platform has edge cases not caught in shadow mode | Fills fail for real users on that platform | Per-platform feature flags allow instant rollback. Monitor fill success rate closely for the first 48 hours after each cutover. |
| Workday's complexity causes persistent issues | The highest-volume platform is the last to cut over, delaying full rollout | Start Workday-specific tuning early (during Phase 5, use shadow mode data to fix Workday issues). Consider a Workday-specific sub-phase with dedicated test fixtures. |
| Removing old extractor creates a "no going back" moment | If post-cutover issues are found, rollback becomes expensive | Keep the old extractor code in-tree (but unused) for at least 2 weeks after all platforms are cut over. Only delete after confirming stability. |

### Testing Strategy

- **Unit tests** (Phases 1-4): Every component is testable with mock inputs. No browser, no LLM calls, no network. Target: fast enough to run on every commit.
- **Integration tests** (Phases 1, 3, 5): Require real browser (Playwright) and/or real LLM API calls. Tagged with `@pytest.mark.integration` so they can be excluded from CI fast path.
- **Parity tests** (Phase 5): Compare old and new extractor output field-by-field on real pages. These are the definitive quality gate before cutover.

### Cost Tracking

The LLM Grouper call must be tracked against the per-job budget (`GH_MAX_BUDGET_PER_JOB`). Use the existing `config/models.py` cost estimation infrastructure. At ~$0.001 per observation and ~5 observations per application, the grouping LLM adds ~$0.005 per job -- well under the $0.50 default budget.

### Logging

All new code uses `structlog` (per CLAUDE.md conventions). Key log events:

- `observer.extract` — Layer 1 completed, with `element_count` and `extraction_ms`
- `observer.interpret` — Layer 2 completed, with `interpretation_ms` and `grouping_ms`
- `observer.diff` — Layer 3 completed, with `fields_added`, `fields_removed`, `values_changed`
- `observer.shadow_mismatch` — Shadow mode detected a discrepancy (Phase 5)
- `grouper.llm_call` — LLM grouping call, with `provider`, `latency_ms`, `token_count`
- `grouper.llm_fallback` — LLM call failed, using heuristic fallback, with `error`
